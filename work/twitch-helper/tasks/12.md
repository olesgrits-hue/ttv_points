# Task 12: Code Audit

## Required Skills

Before starting, load: /skill:code-reviewing — [SKILL.md](../../../.claude/skills/code-reviewing/SKILL.md)

## Description

Full-feature code quality audit of the entire twitch-helper implementation. This task runs after all implementation waves are complete (Tasks 1–11) and reviews the whole codebase holistically — not file by file, but as a system.

The audit focuses on four dimensions:

1. **Singleton lifecycle** — TwitchClient, Queue, OverlayServer, and ConfigStore must each be created exactly once in `src/main/index.ts` and shared via injection, never re-instantiated by consumers.
2. **IPC contract consistency** — renderer→main channels (auth, slot CRUD, toggle) and main→renderer pushes (`twitch:status`, `auth:logout`, log entries) must be consistently defined, typed, and validated on the main-process side.
3. **Error propagation** — action errors must propagate: handler catches → `cancelRedemption` called → IPC log entry emitted with `errorMessage`. No silent swallows, no unhandled promise rejections.
4. **TypeScript strictness** — no `any`, no suppressed errors, no missing return types on public APIs; shared types in `store/types.ts` used consistently across layers.

The output is a written audit report delivered as the agent's final message (not a file). The report flags concrete violations with file+line references and assigns severity (critical / warning / suggestion).

## What to do

1. Read all source files across `src/main/`, `src/renderer/`, and `src/overlay/` — every `.ts`, `.tsx`, and `.html` file produced by Tasks 1–11.

2. Audit singleton lifecycle:
   - Verify each of the four singletons (TwitchClient, Queue, OverlayServer, ConfigStore) is created in `main/index.ts` and nowhere else.
   - Check that action handlers (MaskAction, MediaAction) receive singletons via constructor/function parameters, not by importing and instantiating themselves.

3. Audit IPC contracts:
   - List all `ipcMain.handle` / `ipcMain.on` registrations and all `ipcRenderer.invoke` / `ipcRenderer.on` usages.
   - Verify every channel used in the renderer has a corresponding handler in main.
   - Verify all IPC inputs in main handlers are validated before use (Decision 9 — security hardening).
   - Check that `contextIsolation: true`, `sandbox: true`, `nodeIntegration: false` are set on the BrowserWindow.

4. Audit error propagation:
   - Trace the path: redemption event → Queue → dispatcher → MaskAction or MediaAction → fulfill/cancel → IPC log.
   - Verify every `catch` block either rethrows, calls `cancelRedemption`, or emits a log entry — not just `console.error`.
   - Check that the Queue handles action-handler rejections and does not stall (next item still processes after an error).

5. Audit TypeScript strictness:
   - Check for `any` usages, especially in IPC message handlers and API response parsing.
   - Verify `tsconfig.json` has `"strict": true`.
   - Check that `Slot`, `Config`, `LogEntry`, `MediaRegistryEntry` types from `store/types.ts` are used consistently — no ad-hoc inline object shapes for known types.

6. Cross-check architecture decisions:
   - Decision 3: OverlayServer binds to `127.0.0.1`, not `0.0.0.0`.
   - Decision 4: `keytar` used for access/refresh tokens; no tokens in `config.json` or any plaintext file.
   - Decision 5: ConfigStore uses `PORTABLE_EXECUTABLE_DIR` with `!app.isPackaged` fallback to `process.cwd()`.
   - Decision 2: `robotjs` used for hotkey simulation; no direct `SendInput` FFI.
   - Decision 8: single global `removeMaskHotkey` field in config used by MaskAction, not per-slot.

7. Write the audit report as your final response. Structure:
   - **Summary** — overall assessment (pass / pass with warnings / fail).
   - **Singleton lifecycle** — findings or "OK".
   - **IPC contracts** — findings or "OK".
   - **Error propagation** — findings or "OK".
   - **TypeScript strictness** — findings or "OK".
   - **Architecture decisions** — per-decision status.
   - **Action items** — prioritized list: critical (must fix before deploy) / warning / suggestion.

## Details

### Audit focus areas

**Singleton lifecycle (Architecture § Shared resources table)**

The tech-spec explicitly defines ownership and instance count for four singletons. Common violations to look for:
- `new TwitchClient()` called inside an action handler or IPC handler rather than at bootstrap.
- `ConfigStore` imported as a module singleton via a top-level `const store = new ConfigStore()` in a non-index file — acceptable only if module system guarantees single load; flag if multiple instances are possible.
- Queue constructed inside the dispatcher rather than passed in.

**IPC contract consistency**

Expected renderer→main channels (from tech-spec Tasks 4, 5, 7, 10, 11):
- `auth:login` — start OAuth flow
- `auth:logout` (push from main) — force auth screen
- `slots:list`, `slots:create`, `slots:update`, `slots:delete`, `slots:toggle`
- `twitch:status` (push from main) — connection state
- `log:entry` (push from main) — event log row

Verify that preload script (or contextBridge) exposes exactly these channels and nothing broader. `ipcRenderer.send` with arbitrary channel names from renderer-side code is a security red flag.

**Error propagation path**

Full expected chain for a failed media action:
```
Queue.process() calls MediaAction.execute()
  → MediaAction throws / rejects
  → Queue catch: calls twitchClient.cancelRedemption(redemptionId)
  → Queue catch: emits mainWindow.webContents.send('log:entry', { status: 'error', errorMessage })
  → Queue: advances to next item (does NOT break the loop)
```

Any deviation — missing cancel, missing log emit, or queue stall — is a critical finding.

**TypeScript strictness**

Focus on:
- `ipcMain.handle('channel', async (event, args) => { ... })` — `args` typed as `unknown` and validated, not cast to a specific type without check.
- Twitch API response parsing — response body typed via interface, not `any`.
- `keytar.getPassword` return is `string | null` — null case handled before use.
- `fs.readdirSync` return used for folder contents — items filtered to supported media extensions.

**Architecture decisions cross-check**

Decision 3 (overlay bound to 127.0.0.1): look for `app.listen(7891)` — should be `app.listen(7891, '127.0.0.1')`.
Decision 4 (keytar): search for any write of `accessToken` or `refreshToken` to `config.json` or any `.json` file.
Decision 5 (PORTABLE_EXECUTABLE_DIR): ConfigStore base path logic — must use env var with isPackaged guard.
Decision 2 (robotjs): mask action must use `robot.keyTap(key, modifiers)`, not any FFI or `child_process.exec('xdotool')`.
Decision 8 (global remove-mask hotkey): `MaskAction` must read `config.removeMaskHotkey`, not a per-slot field.

### What to output

The audit report is your final assistant message. Do not write it to a file. Structure it clearly using markdown headings. Be specific — every finding must include the file path and line number (or function name if line is uncertain). Avoid vague statements like "error handling could be improved"; instead write "src/main/queue/index.ts `processNext()`: catch block calls `console.error` only — missing `cancelRedemption` call and `log:entry` IPC emit (critical)".
