# Task 7: FIFO Queue & Action Dispatcher

## Required Skills

## Description

Implements the FIFO redemption queue and action dispatcher that form the core processing pipeline for channel point redemptions. The queue ensures redemptions are handled one at a time in strict arrival order — concurrent redemptions do not overlap or race. The dispatcher looks up the slot matching each redemption's `rewardId`, validates it (exists and enabled), and routes to the appropriate action handler (`MaskAction` or `MediaAction`). On any invalid state (slot not found or slot disabled), the dispatcher cancels the redemption via the Twitch API and emits an error log entry. The queue also participates in the Twitch disconnect/reconnect lifecycle: it pauses processing when `TwitchClient` emits `'disconnected'` and resumes transparently when `'connected'` is emitted — in-memory items are preserved throughout.

This sits at the center of the data flow: `TwitchClient` enqueues events into `Queue`; `Dispatcher` reads `ConfigStore` to resolve slots; results go to `MaskAction` or `MediaAction` (Tasks 8 and 9). The queue singleton is bootstrapped in `main/index.ts`.

## What to do

1. **Create `src/main/queue/types.ts`** — shared types for the queue subsystem:
   - `RedemptionEvent` interface (fields: `id`, `rewardId`, `rewardTitle`, `userLogin`, `userDisplayName`, `redeemedAt`)
   - `ActionHandler` interface: `execute(slot: Slot, redemption: RedemptionEvent): Promise<void>`

2. **Create `src/main/queue/dispatcher.ts`** — the Action Dispatcher:
   - Class with constructor receiving `configStore: ConfigStore` and `twitchClient: TwitchClient` (dependency injection for testability)
   - `dispatch(redemption: RedemptionEvent): Promise<void>` — single public method
   - Slot lookup: `configStore.getSlots().find(s => s.rewardId === redemption.rewardId)`
   - Slot not found → `cancelRedemption` + emit error log entry
   - Slot found but `!slot.enabled` → `cancelRedemption` + emit error log entry
   - Slot found and enabled → route to `MaskAction.execute(slot, redemption)` (type `'mask'`) or `MediaAction.execute(slot, redemption)` (type `'media'` or `'meme'`)
   - Wrap action call in try/catch: uncaught error → cancel + error log (no crash)
   - Action handlers imported as stubs satisfying `ActionHandler` interface (Tasks 8/9 fill them in)

3. **Create `src/main/queue/index.ts`** — the FIFO Queue class:
   - Internal `_items: RedemptionEvent[]` — in-memory FIFO array
   - Internal `_processing: boolean` — prevents concurrent runs
   - Internal `_paused: boolean` — set by `pause()`/`resume()`
   - `enqueue(redemption: RedemptionEvent): void` — max-5-slots check first (read `configStore.getSlots().length`; if ≥ 5: cancel + error log, do not push); push to `_items`; call `_processNext()`
   - `_processNext()` (private): if paused or empty or already processing → return; set `_processing = true`; shift item; await `dispatcher.dispatch(item)`; recursively call `_processNext()`; set `_processing = false`
   - `pause()` — sets `_paused = true`; items queued while paused are preserved
   - `resume()` — sets `_paused = false`; calls `_processNext()` if items pending
   - `size(): number` — returns `_items.length` (for test introspection)
   - Subscribe to `twitchClient` events in constructor: `'disconnected'` → `pause()`; `'connected'` → `resume()`

4. **Document bootstrap integration point** — add a `// TODO(bootstrap): in main/index.ts create Queue singleton, pass ConfigStore + TwitchClient` comment; do not modify `main/index.ts` (that belongs to the bootstrap task)

## Details

**Files:**

- `src/main/queue/types.ts` — new file; `RedemptionEvent` and `ActionHandler` interfaces; `RedemptionEvent.id` is the redemption ID used for fulfill/cancel API calls
- `src/main/queue/dispatcher.ts` — new file; constructor injects `configStore` and `twitchClient`; IPC log emit pattern: `BrowserWindow.getAllWindows()[0]?.webContents.send('log:entry', logEntry)`; `LogEntry` fields: `id` (uuid v4), `timestamp` (new Date()), `viewerName` (redemption.userDisplayName), `rewardTitle` (slot?.rewardTitle ?? redemption.rewardTitle), `status: 'error'`, `errorMessage`
- `src/main/queue/index.ts` — new file; uses Node.js `EventEmitter` as base class (emit `'paused'`/`'resumed'` for test observability); recursive `_processNext()` pattern (not while-loop) — correct for low-frequency channel point events

**Dependencies:**
- `uuid` — generate `LogEntry.id`
- `electron` — `BrowserWindow` for IPC log emit
- `TwitchClient` (Task 5) — provides `cancelRedemption(redemptionId: string)` and EventEmitter interface
- `ConfigStore` (Task 2) — provides `getSlots(): Slot[]`
- `MaskAction` (Task 8) and `MediaAction` (Task 9) — import as stubs now; implement `ActionHandler` interface; this task does not require Tasks 8/9 to be complete

**Edge cases:**
- `resume()` on empty queue → no-op, no error
- `pause()` called twice → idempotent
- `cancelRedemption` itself throws → wrap in try/catch; log failure but do not crash queue
- Max-5 check uses `configStore.getSlots().length` (configured slot count), NOT queue depth
- Enqueue while paused → item queued, processed after `resume()` in original order
- Enqueue during active processing → item pushed to tail; `_processNext()` picks it up after current item finishes

**Implementation hints:**
- Use dependency injection (constructor params) throughout — no module-level singletons inside queue/dispatcher files; singleton creation belongs in `main/index.ts`
- For unit tests: mock `configStore` with `{ getSlots: jest.fn(() => mockSlots) }`; mock `twitchClient` as a jest mock extending EventEmitter so `.emit('disconnected')` works in tests
- `_processNext()` recursive call must happen in a `finally` block or after await resolves — ensure it always fires even when dispatcher throws (error is caught before reaching `_processNext`)
